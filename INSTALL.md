# Cross-AI Progress Bridge v3.9

每個口令開始時，本協定新增發起的讀寫只限目前專案與 manifest 明列路徑；這不能撤回平台已預先載入的內容，也不能推翻更高優先規則。若已載入的高層規則要求額外讀取，先列精確路徑與用途並取得使用者同意，不得聲稱能以本協定阻擋。接著用 lstat 或等效 filesystem metadata 驗證固定路徑 AI_SYSTEM/manifest.md；從專案根目錄到來源與目的地的每個既有路徑元件都不得是 link／junction／reparse，canonical path 必須仍在專案內，無法可靠判定也要停止。通過才讀 manifest；再以同樣方式驗證 protocol、progress_root、legacy_rules 及非空的 verification_task 都是無磁碟代號、無 .. 的相對路徑。每次 read／create／write／move／delete 前，對來源、目的地及最近既有祖先重做同一檢查；新目錄逐層建立，每建立一層就立即重查。

每個既有普通檔在讀取、備份、修改或移動前，還要取得 hard-link count 並確認正好為 1；大於 1 或無法可靠取得就停止。每個新建普通檔建立後也要立即確認 hard-link count 為 1。fresh 模式的 live task set 只包含 `progress_root` 直層的普通 `.md` 任務檔，排除 verification_task，也不掃描 `archive/`、`conflicts/` 或任何其他子目錄。相容模式只使用獲准且已記錄在 legacy_rules 的 live task 來源，排除其封存／衝突來源及 verification_task；沒有明定來源就停止詢問，不得全專案猜測。

修改任何既有任務檔或 manifest 都禁止原地覆寫：先把原檔備份到 `AI_SYSTEM/backups/runtime/`，同名加時間與流水號且不覆蓋舊檔，再在原檔同一目錄以 create-only 建立唯一 `.tmp` 候選檔並完整讀回；緊接原子替換前，重新讀取原檔並確認完整內容雜湊與預期舊值相同，任務 revision 也相同。只有作業系統／檔案工具能保證同一檔案系統的 atomic replace 時才能替換；無法保證、原檔已變或任何驗證失敗就保留原檔，將候選內容移到 conflicts 或保留 `.tmp` 後停止。原子替換後再讀回驗證。多檔狀態轉移除上述單檔規則外，還必須使用下述 journal，不得靠操作順序猜測。
